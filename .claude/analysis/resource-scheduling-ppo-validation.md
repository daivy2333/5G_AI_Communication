## 2. PPO 执行链路

### 调用链

```text
collect_experiment_data.py::collect_resource_scheduling
  -> PPOScheduler(env_config, agent_config)
  -> PPOScheduler.train(timesteps, log_interval)
  -> stable-baselines3 PPO.learn(...) if stable_baselines3 is available
  -> otherwise _train_ppo(...)
  -> scheduler.training_stats
  -> resource_scheduling_reward.csv / resource_scheduling_train.log / real_ppo_*.{png,pdf}
```

| 接口 | 定义位置 | 用途 |

## 3. 环境与指标定义

资源调度环境状态是每个用户的归一化信道质量和队列长度，形状为 `num_users * 2`。动作空间是 `MultiDiscrete([num_users] * num_resource_blocks)`，即每个资源块分给一个用户。

环境每一步做三件事：

1. 根据动作统计每个用户拿到的 RB。
2. 用当前信道增益估算速率，再计算传输比特数。
3. 更新队列，并返回 `throughput`、`fairness`、`energy_efficiency`。

## 4. 风险点

### 4.1 stable-baselines3 与自实现 PPO 统计路径不同

`PPOScheduler.train()` 在有 stable-baselines3 时走 `self.model.learn(...)`，依赖 `ProgressCallback` 把指标写入 `training_stats`。没有 stable-baselines3 时走 `_train_ppo()`。T06 验收必须看输出文件，而不是只看命令是否结束。

### 4.2 自实现 PPO 的动作维度是弱点

环境动作要求长度等于 `num_resource_blocks`，但自实现 `PPONetwork.get_action()` 采样的是单个 categorical 动作；`SchedulingEnvironment.step()` 会用 `action[:num_resource_blocks]` 解析动作。若运行环境没有 stable-baselines3，自实现分支存在动作形状不匹配风险。当前机器若安装 stable-baselines3，会优先规避这个风险。

### 4.3 指标单位要保持一致

环境 `info["throughput"]` 来自 `total_throughput`，实验脚本写入列名为 `throughput_Mbps`。文档和论文集成时应以脚本输出字段为准，不从源码注释推断单位。

## 5. 建议执行顺序

1. 先运行只含调度的收集命令，缩小 T06 范围。
2. 检查 `summary.json` 是否新增 `resource_scheduling`。
3. 检查 CSV、日志和 3 张图是否生成。
4. 若失败，先判断是否走了自实现 PPO；若是动作维度问题，优先安装/启用 stable-baselines3 或单独修正自实现动作输出。
