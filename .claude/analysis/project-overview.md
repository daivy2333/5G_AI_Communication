## 1. Repository Shape

This repository has a two-layer structure:

| Layer | Entry | Purpose |
|------|-------|---------|
| Paper layer | `conference_101719.tex:17` | IEEE conference paper for closed-loop 5G channel estimation, signal recognition, and PPO scheduling |
| Experiment layer | `5G_AI_Communication/main.py:466` | Python implementation and experiment driver for the paper's modules |

The paper layer is not only a generic LaTeX template now. `conference_101719.tex` defines the paper title and imports five authored sections:

```text
conference_101719.tex
  -> 0-intro.tex
  -> 1-bridge-gaps.tex
  -> 2-methodology.tex
  -> 3-experiments-results.tex
  -> 4-discussion-conclusion.tex
```

The Python layer implements the system described by the paper: AI channel estimation, signal detection and modulation recognition, resource scheduling, OFDM simulation, metrics, visualization, and experiment data collection.

## 2. Core Execution Flow

`5G_AI_Communication/main.py` is the unified CLI entry. It exposes four modes: `demo`, `full`, `channel_estimation`, `signal_detection`, and `resource_scheduling`.

```text
main()
  -> run_demo_mode() or run_full_mode() or run_single_module()
    -> run_channel_estimation_demo()
       -> ChannelEstimationTrainer.run_full_training()
    -> run_signal_detection_demo()
       -> DetectorTrainer.train()
       -> DetectorTrainer.evaluate()
    -> run_resource_scheduling_demo()
       -> create_scheduler("ppo")
       -> scheduler.train()
    -> generate_comparison_report()
       -> results/comparison_report.json
```

The main driver is suitable for demonstrations and quick integration checks. The stronger paper-data boundary is `docs/experiments/collect_experiment_data.py`, which exports structured CSV, logs, figures, and `summary.json` for real experiment reporting.

## 3. Module Boundaries

| Module | Key files | Responsibility |
|--------|-----------|----------------|
| Channel estimation | `channel_estimation/models.py`, `channel_estimation/trainer.py`, `channel_estimation/data_generator.py` | Transformer-based AI-ChannelNet, LS/MMSE/LMMSE baselines, synthetic 5G channel data |
| Signal detection | `signal_detection/detector.py` | CNN-BiLSTM binary signal/bit detection |
| Modulation recognition | `signal_detection/recognizer.py`, `signal_detection/modulator.py` | Five-class BPSK/QPSK/16QAM/64QAM/256QAM recognition and IQ sample generation |
| Resource scheduling | `resource_scheduling/agent.py`, `resource_scheduling/environment.py` | PPO/DQN-style scheduler and multi-user resource-block environment |
| Link simulation | `simulation/transceivers.py` | OFDM transmitter/receiver primitives |
| Metrics | `utils/metrics.py` | NMSE, recognition, scheduling, and communication metrics |
| Visualization | `visualization/dashboard.py` | Flask dashboard for constellation, spectrum, channel, and metric monitoring |

Configuration is centralized in `config.py`: system constants begin at `SystemConfig`, model hyperparameters at `ChannelEstimationConfig` and `SignalDetectionConfig`, and PPO/environment parameters at `ResourceSchedulingConfig`.

## 4. Paper-Implementation Relationship

The paper's methodology maps directly to code concepts:

| Paper concept | Implementation anchor |
|--------------|------------------------|
| 3.5 GHz, 100 MHz, 2048 FFT, 512 CP, 1200 subcarriers | `config.py:9` |
| AI-ChannelNet Transformer, 4 layers, 8 heads, 128 embedding | `config.py:32` and `channel_estimation/models.py` |
| CNN-BiLSTM SignalNet | `config.py:57` and `signal_detection/detector.py:71` |
| Five-class modulation recognition | `signal_detection/recognizer.py:48` and `signal_detection/modulator.py` |
| PPO scheduler with throughput/fairness/energy reward | `config.py:85`, `resource_scheduling/agent.py:133`, `resource_scheduling/environment.py` |
| Structured experiment outputs | `docs/experiments/collect_experiment_data.py:72`, `:157`, `:222`, `:331` |

The paper result text currently mixes older summary values and newer real-training values. For example, `3-experiments-results.tex` still preserves a cautious channel result where AI-ChannelNet reaches `0.04 dB`, while `docs/PERFORMANCE.md` reports real-training AI-ChannelNet around `-20.49 dB` and final summary `-20.42 dB`. Future paper edits should choose one authoritative evidence set, preferably the real experiment outputs documented in `docs/PERFORMANCE.md`.

## 6. Main Risks and Maintenance Notes

1. The root repository name and assets still look like a generic conference LaTeX template, but the active content is a 5G AI paper plus Python experiment project.
2. `conference_101719.tex` and section files should be treated as the paper source of truth, while `5G_AI_Communication/docs/PERFORMANCE.md` and experiment outputs should be treated as the quantitative evidence source.
3. `main.py` is an integration/demo entry, not necessarily the best source for publication-grade metrics.
4. The OpenSpec and Claude state live under `5G_AI_Communication/`, not at the root. Future project-memory updates should target that subtree unless the root paper itself gets a separate OpenSpec setup.
5. CodeGraph indexing at root sees the Python subtree; LaTeX structure still needs direct file reading because CodeGraph does not parse TeX here.
